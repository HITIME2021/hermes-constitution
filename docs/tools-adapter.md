# Tools Adapter

Tools Adapter defines how Hermes wraps external tools so they can be invoked
through a consistent policy, evidence, and artifact intake interface.

This document is tool-agnostic. Specific tools may appear as examples, but the
adapter contract is based on usage mode and effects, not the tool brand.

## Core Rule

<!-- snapshot:block id="tools-adapter" section="Tools Adapter" priority="82" -->
Hermes must invoke external tools through a declared Tools Adapter entry.

```text
Tools Layer decides what class a tool invocation belongs to.
Tools Adapter decides how the tool is invoked, scoped, captured, and mapped.
Artifact Intake Gate decides whether frontend artifacts may influence execution.
Provider Adapter policy decides whether provider work may execute.
```

A Tools Adapter entry must declare:

```yaml
tool_adapter:
  tool_name: example-tool
  execution_plane: WSL
  invocation:
    command: example
    args_template:
      - subcommand
  supported_modes:
    - read_only_inspection
    - artifact_generation
    - backend_effect
  default_classification: backend_tool_until_mode_is_proven_artifact_only
  allowed_read_only_commands:
    - "--help"
    - "version"
    - "check"
  artifact_outputs:
    - spec.md
    - plan.md
    - tasks.md
    - checklist.md
  prohibited_without_execution_request:
    - project_mutation
    - dependency_install
    - provider_routing
    - workflow_execution
    - self_upgrade
    - memory_writeback
```

Read-only inspection commands may be used to verify tool availability,
versions, capabilities, and help text. Their output is environment evidence
only. Tool discovery output must not automatically modify Hermes Provider
Policy, approve new providers, or expand execution scope.

Project read-only does not always mean environment no-effect. A command may
leave the target project unchanged while still using network access, writing a
tool cache, building a temporary package, or updating runtime metadata. Hermes
must record these runtime effects separately from project mutation.

For `uvx`-based tool invocation, Hermes must treat the invocation as
project-read-only only when no target project files are changed. Network use,
package build, install cache, or tool cache behavior must be captured as
runtime environment evidence when relevant.

Tool evidence intended for audit or long-term operator review should be
captured as UTF-8 text. If output encoding is degraded, garbled, or lossy,
Hermes must mark the evidence as encoding-degraded and should regenerate the
evidence with explicit UTF-8 settings before relying on it for memory,
release notes, or policy updates.

Artifact generation mode may produce planning artifacts. Those artifacts remain
input evidence and must pass Artifact Intake Gate before they influence live
execution.

Backend effect mode may create files, install templates, run workflows, update
integrations, mutate runtime state, or perform other observable side effects.
Backend effect mode requires an `ExecutionRequest`, declared scope, approval
state when needed, stop conditions, and audit-grade evidence.

Hermes must reclassify every invocation. A tool that is safe in read-only mode
does not become safe for initialization, workflow, upgrade, integration, or
project mutation.

Example adapter entry:

```yaml
tool_adapter:
  tool_name: spec-kit/specify
  execution_plane: WSL
  invocation:
    command: uvx
    args_prefix:
      - "--from"
      - "git+https://github.com/github/spec-kit.git"
      - "specify"
  verified_capability_evidence:
    cli_version: "0.12.16.dev0"
    python: "3.11.15"
    platform: "Linux x86_64"
  runtime_effects:
    project_mutation: false_for_read_only_inspection
    possible_environment_effects:
      - network_access
      - uv_cache_write
      - temporary_package_build
    evidence_required: true
  read_only_inspection:
    - "--help"
    - "version"
    - "check"
    - "init --help"
  artifact_mode:
    class: frontend_tool
    authority: input_evidence_only
    requires: Artifact Intake Gate
  init_or_workflow_mode:
    class: backend_tool
    requires: ExecutionRequest
    recommended_scope: isolated_workspace_or_explicit_project_scope
  forbidden_without_explicit_approval:
    - "init --here"
    - "init ."
    - "init --here --force"
    - "self upgrade"
    - "workflow run"
```

Tool integration files generated by a tool, such as Codex or CodeBuddy command
templates, do not change Hermes Provider Policy. Provider roles remain governed
by Project Policy, Provider Adapter policy, and operator approval.

## Adapter Responsibilities

A Tools Adapter should:

- construct the exact command invocation
- declare the execution plane
- classify the invocation mode
- bind allowed and forbidden scopes
- capture stdout, stderr, exit code, generated files, and diagnostics
- map artifacts to Hermes artifact types
- record tool version and capability evidence when available
- surface stop conditions without retry loops that waste provider tokens

## Non-Goals

Tools Adapter does not make tool outputs authoritative.

Tools Adapter does not approve providers discovered by a tool.

Tools Adapter does not replace Artifact Intake Gate, Provider Adapter policy,
Review Gate, Human Intervention policy, or Memory Writeback approval.
